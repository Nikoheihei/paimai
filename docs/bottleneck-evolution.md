# 压测瓶颈演进

本文记录 `paimai` 在压测过程中发现瓶颈、定位根因、完成优化并验证结果的主要过程。完整压测数据见 [压测结果](performance-test-results.md)。

## 迭代概览

| 阶段 | 问题/改动 | 结果 |
| --- | --- | --- |
| 迭代 0 | 初始热点压测，200 人抢 1 个拍卖 | 47% HTTP 5xx，成功出价约 30/s |
| 迭代 1 | 乐观锁冲突从 500 改为 409，并补充 `rejectCode` | `system_error` 归零，业务拒绝可观测 |
| 迭代 2 | Redis Lua 做价格预过滤 | 低价无效请求提前拦截，MySQL 事务压力下降 |
| 迭代 3 | Redis 状态统一由 Consumer 更新 | 消除 PlaceBid 与 Consumer 双写不一致风险 |
| 迭代 3.5 | Outbox 轮询从 2s/50 条优化到 100ms/500 条 | pending 从 40,758 降到接近 0，事件延迟降到毫秒级 |
| 迭代 3.6 | WebSocket 广播改为三段式无锁化 | 2000 连接 P99 3.2s 问题消失，支持 10,000 长连接零断线 |
| 迭代 4 | 全量压测验证 | 多房间峰值约 1,900 成功出价/s，`system_error=0` |

## 关键发现

### 1. 单拍卖热点是业务竞争，不是服务器性能不足

同样 200 个用户，只改变拍卖数量：

| 实验 | 用户数 | 拍卖数 | 成功出价/s | HTTP 2xx | 5xx+超时 |
| --- | --- | --- | --- | --- | --- |
| A | 200 | 1 | 30/s | 50.8% | 47.2% |
| B | 200 | 200 | 2,387/s | 100% | 0% |

结论：瓶颈不是 Gin、Redis、MySQL 整体处理能力或网络，而是大量请求同时竞争同一条 `auctions` 行时产生的乐观锁冲突。

### 2. 错误码必须区分业务拒绝和系统异常

初始版本把乐观锁冲突返回为 HTTP 500，导致压测中 `system_error` 虚高。修正后：

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| `system_error` | 47.2% | 0% |
| 乐观锁冲突 | HTTP 500 | HTTP 409 |
| 拒绝原因 | 混在系统错误中 | 通过 `rejectCode` 精确分类 |

这一步没有显著提升吞吐，但让系统健康度和业务竞争失败可以被准确区分。

### 3. MySQL 是最终事实源，Redis 只做预过滤和派生状态

优化后保持两条原则：

- MySQL 仍然是出价、结算和状态变更的最终事实源。
- Redis 只承担价格预过滤、排行榜、状态缓存和实时广播所需的派生数据。

最终写入链路收敛为：

```text
MySQL transaction
  -> Outbox event
  -> Redis Stream
  -> Consumer updates Redis state and broadcasts WebSocket events
```

这避免了 PlaceBid 和异步 Consumer 同时写 Redis 造成的状态分叉风险。

### 4. Outbox 和 Consumer 参数决定事件实时性

在约 1,900 成功出价/s 的压测下，原来的 `2s` 轮询和每批 `50` 条会快速堆积事件。优化后：

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| Outbox 轮询间隔 | 2s | 100ms |
| 每批处理数量 | 50 | 500 |
| Outbox pending | 40,758 | 接近 0 |
| 事件到广播延迟 | 秒级 | 毫秒级 |

### 5. WebSocket 广播瓶颈来自锁粒度

原广播模型在 Hub 全局锁下遍历客户端，连接数上升后会阻塞不同房间的广播。优化为三段式：

1. `Hub.mu` 只负责查找 room 指针。
2. `Room.mu` 只负责拷贝 client 列表。
3. 实际发送阶段无锁，并使用非阻塞发送。

慢客户端保护策略：

- `client.send` 提供短时缓冲。
- 缓冲满时直接丢弃并标记连接死亡，不阻塞整个房间广播。

结果：广播链路从 2000 连接 P99 3.2s，优化到 10,000 连接稳定运行、零断线。

## 最终对比

| 指标 | 初始状态 | 优化后 |
| --- | --- | --- |
| `system_error` | 47% | 0% |
| 单拍卖成功出价/s | 30/s | 73/s |
| 多房间成功出价/s | 未验证 | 约 1,900/s |
| 拒绝原因 | 500 掩盖一切 | 409 + `rejectCode` 精确分类 |
| Redis 写入路径 | 多处写入，存在不一致风险 | Consumer 统一写入 |
| Outbox pending | 最高 40,758 | 接近 0 |
| WebSocket 支撑能力 | 2000 连接广播延迟明显 | 10,000 连接零断线 |

## 结论

`paimai` 当前的核心瓶颈不是基础设施吞吐，而是单拍卖热点竞争下的业务行锁/乐观锁冲突。多房间、多拍卖场景能有效摊薄热点，系统整体吞吐和实时广播链路表现稳定。

后续若要继续提升单拍卖热点能力，可以考虑：

- 批量合并出价或引入队列化单拍卖撮合。
- 对同一拍卖做内存串行化竞价执行器。
- 将热点拍卖状态从 MySQL 行级乐观锁迁移到更适合高频写入的状态机模型。
