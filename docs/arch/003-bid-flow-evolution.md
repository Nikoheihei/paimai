# 拍卖系统性能测试与优化总结

## 一、测试目标

验证直播拍卖系统在高并发场景下的稳定性、吞吐能力及瓶颈位置。

技术栈：

* Go
* Gin
* MySQL 8
* Redis 7
* WebSocket
* Outbox Pattern

测试环境：

* Docker 单实例部署
* 本地压测
* MySQL 作为 Truth Source
* Redis 作为缓存与广播组件

---

# 第一轮迭代：


## 二、问题发现:第一轮压测：单拍卖热点测试

场景：

```text
200人同时竞争同一个拍卖
```

压测结果：

| 指标       | 数值     |
| -------- | ------ |
| 成功出价/s   | 30     |
| HTTP 2xx | 50% 左右 |
| 5xx+超时   | 47%    |
| P95      | 54ms   |

现象：

* 大量请求返回 HTTP 500
* 成功出价吞吐急剧下降
* 用户并发增加后成功率持续下降

---

## 三、问题定位

初步怀疑：

```text
MySQL 乐观锁竞争
```

进一步分析日志发现：

```sql
UPDATE auctions
SET version = version + 1
WHERE id = ? AND version = ?
```

大量请求：

```text
RowsAffected = 0
```

即：

```text
乐观锁冲突
```

但系统将其返回：

```text
HTTP 500
```

实际上：

```text
这是业务竞争失败
不是系统异常
```

```text
200人竞争200个拍卖
```

结果：

```text
成功出价/s：2407
HTTP 2xx：100%
系统错误率：0%
```

结论：

```text
服务器不是瓶颈
Redis不是瓶颈
MySQL整体能力不是瓶颈

瓶颈来自：
单个 Auction 的热点竞争
```

## 五、优化方案

### 优化1：修正错误码

优化前：

```text
乐观锁冲突 → HTTP 500
```

优化后：

```text
乐观锁冲突 → HTTP 409 Conflict
```

目的：

```text
区分业务失败与系统失败
```

---

### 优化2：Redis价格预过滤

优化前：

```text
所有请求进入MySQL
```

优化后：

```text
读取Redis当前价格

bidAmount <= currentPrice

直接业务拒绝
```

流程：

```text
Redis预校验
↓
MySQL乐观锁
↓
Outbox
↓
Consumer更新Redis
↓
WebSocket广播
```

原则：

```text
MySQL仍然是Truth Source
Redis只负责预过滤
```

---

### 优化3：统一Redis写入路径

优化前：

```text
PlaceBid写Redis

Consumer也写Redis
```

存在：

```text
双重写入
状态不一致风险
```

优化后：

```text
Redis只由Consumer更新
```

统一数据流：

```text
MySQL事务成功
↓
Outbox
↓
Consumer
↓
Redis
↓
WebSocket
```

## 六、优化效果

### 单拍卖热点场景

优化前：

| 指标     | 数值    |
| ------ | ----- |
| 成功出价/s | 30.2  |
| 出价接受率  | 28.7% |
| 5xx+超时 | 47.2% |

优化后：

| 指标     | 数值   |
| ------ | ---- |
| 成功出价/s | 33.2 |
| 出价接受率  | 1.7% |
| 5xx+超时 | 0%   |

结论：

```text
成功吞吐变化不明显

但：
系统错误率从47%降至0%

大量低价请求被Redis提前拦截
```
