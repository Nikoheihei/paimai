# 「4-3-2-1」AI 开发范式落地 — 实时竞拍大师项目

> 本文档将"AI 为中心的开发新范式"理论，提炼为**可直接在本项目中执行**的 4-3-2-1 行动框架。
>
> 核心一句话：**把开发从"人写代码"变成"人设计认知流水线，AI 跑全流程，人只守边界偏差"。**

---

## 一、4 个核心原则（本项目的底线）

### 原则 1：AI 中心，人退边界

**含义**：AI 是编码、测试、修复的主体。人只做两件事——定义目标、仲裁 AI 搞不定的偏差。

**在本项目中的落地**：

| 当前模式（要淘汰的） | 目标模式 |
|---|---|
| 人写 Go handler / service / repository | 人写产线配置文件（prompt + 校验规则），AI 生成所有代码 |
| 人 Review AI 生成的代码 | 自动化测试关卡代替人 Review |
| 人补 AI 漏写的边界条件 | AI 自测失败后自动修复，修复不了才升级给人 |

**具体实践**：
- 每次开发新模块（如 WebSocket Hub、结算任务），不直接写代码，先写"产线定义"：
  - 接口契约（proto / OpenAPI / 接口签名）
  - 验证规则（单测断言、边界条件列表）
  - 知识参考（现有类似代码位置、模式示例）
- 之后让 AI 全量生成，自动跑测试，不通过就自动迭代。

### 原则 2：确定性裁判前置

**含义**：所有 AI 输出必须先过**自动化验证关卡**，不依赖人的主观判断。

**在本项目中的落地**：

```
AI 生成代码
    ↓
[关卡 1] go vet / go build — 编译级验证
    ↓
[关卡 2] go test ./... — 单元测试（已有 19 个测试用例做基线）
    ↓
[关卡 3] 契约校验 — 接口参数类型、响应结构、状态机合法状态对
    ↓
[关卡 4] 规则验证 — 竞拍规则合规（加价幅度、封顶逻辑、延时触发条件等）
    ↓
全部通过 → 合入
任一失败 → AI 自动修复 → 回到关卡 1
```

**当前项目已有但未系统化的裁判资源**：
- 状态机（`internal/statemachine/auction.go`）— 可复用为状态流转的契约校验器
- 17 个单元测试用例 — 可作为回归验证基线
- Go 编译器的类型检查 — 天然的第一个裁判关卡

### 原则 3：闭环优先，拒绝碎片化

**含义**：先跑通端到端的小闭环，再扩范围。不搞"先优化一个函数"这种单点替换。

**在本项目中的落地**：

```
第一闭环 [当前基线]              后端 CRUD + 出价（已完成）
    ↓
第二闭环 [本周目标]              出价 → WebSocket 推送 → 结算 → 订单（正在建设）
    ↓
第三闭环 [下周目标]              移动端竞拍页接真实 API
    ↓
第四闭环 [后续]                  压测、氛围动画、PC 后台完整对接
```

每个闭环的验收标准必须是**可运行的端到端用例**，而非"单元测试通过"。

### 原则 4：分治继承，业务切片

**含义**：按端到端业务价值拆分产线单元，不按"前端/后端/DBA"技术模块拆。

**在本项目中的落地**：

| ❌ 不要这样切 | ✅ 要这样切 |
|---|---|
| 后端任务 / 前端任务 / 数据库任务 | 商品管理与竞拍发布产线（从创建到上架） |
| Controller / Service / Repository | 用户出价闭环产线（从请求到成交） |
| 接口开发 / 接口测试 | 结算与订单产线（从竞拍结束到交易完成） |
| | WebSocket 实时推送产线（从出价到广播到前端） |

每个产线单元包含：**后端接口 + 存储逻辑 + 前端交互 + 测试验证**。

## 二、3 层落地架构

### 第 1 层：最小闭环 — 从"出价闭环"开始（当前就是最佳切入点）

这是你现在就能启动的最小实践单元。竞拍的"用户出价"边界清晰、规则明确、测试可自验证。

**产线定义文件**（新建 `.codepipe/bid-pipeline.yml`）：

```yaml
pipeline: bid-closed-loop
domain: "用户出价 → 原子校验 → 持久化 → 广播事件"

input:
  - "POST /api/auctions/:id/bids 的请求参数规范"
  - "状态机中合法的状态流转图"
  - "竞拍模式规则文档（sudden_death / extension / reserve）"

verification_gates:
  - "go build ./..."                          # 编译关卡
  - "go test ./internal/service/..."          # 已有测试回归
  - "出价金额必须 > 当前价 + 加价步长"         # 业务规则校验
  - "幂等键不能重复使用"                        # 幂等校验
  - "封顶价到达后状态必须为 sold"                # 终态校验
  - "延时模式下结束前 N 秒出价须延长 T 秒"        # 延时规则校验

workflow:
  - step: "生成 bid handler / service 代码"
    check: "go build"
    fix: true
  - step: "生成对应的单元测试"
    check: "go test"
    fix: true
  - step: "生成业务规则测试（封顶/延时/幂等）"
    check: "go test -run TestBid"
    fix: true
  - step: "生成集成测试骨架（需 Redis + MySQL）"
    check: "go test -tags=integration"
    fix: false  # 集成测试失败需人工确认环境

output:
  - "internal/handler/bid.go"
  - "internal/service/bid.go"
  - "internal/repository/bid.go"
  - "internal/service/bid_test.go"
min_pass_rate: 0.95  # 产线良率目标
```

**你的动作**：不碰中间代码，只验收最终结果（`go test` 全绿 + 端到端 Postman 测试通过）。

### 第 2 层：分层递阶控制（适配本项目复杂度）

```
┌──────────────────────────────────────────┐
│  决策层 — 你（架构师/产品负责人）           │
│  职责：定下一个闭环的目标、安全红线、仲裁      │
│  AI 解决不了的极端场景                      │
│  例："流拍后生成的订单应该自动取消还是保留"    │
├──────────────────────────────────────────┤
│  协调层 — AI 产线工程师（未来可培养的角  色）  │
│  职责：调优产线配置文件 prompt、校验规则、     │
│  知识库，监控产线健康度                      │
│  例：发现出价压测时 Lua 脚本偶尔超时 →        │
│  更新知识库加入超时重试策略                    │
├──────────────────────────────────────────┤
│  执行层 — AI（Codex 等模型）                │
│  职责：跑产线生成代码、自测、自修复            │
│  出了错先自己试 3 次修复，修复不了才升级        │
└──────────────────────────────────────────┘
```

### 第 3 层：分工协调总线（本项目的简化版）

由于项目规模不大（单人/双人），不需要消息队列级别的总线。用项目内的约定文件代替：

**`.codepipe/pipeline-bus.sh`**（编排脚本）:

```bash
#!/bin/bash
# 编排多个产线的执行顺序

echo "=== 阶段 1：商品管理产线 ==="
# (手动触发：创建商品时跑)

echo "=== 阶段 2：竞拍发布产线 ==="   
# (手动触发：配置竞拍规则时跑)

echo "=== 阶段 3：出价闭环产线 ==="
# (手动或持续集成触发)

echo "=== 阶段 4：结算订单产线 ==="
# (依赖阶段 3 的竞拍结束事件)

echo "=== 阶段 5：端到端冒烟测试 ==="
# 启动 Docker 容器，跑完整链路
```

**关键设计**：每个产线输出一份 `pipeline-report.json`，包含：
- 生成的文件列表
- 每个关卡的通过/失败明细
- 失败时的 AI 修复尝试日志
- 产线良率（`通过数 / 总数`）

## 三、2 个必须攻克的硬骨头 — 在本项目的落地

### 硬骨头 1：隐性知识蒸馏 — 从"出价规则"开始练手

竞拍系统里有很多"专家才懂"的隐性规则，比如：

| 隐性知识 | 当前状态 | 蒸馏方法 |
|---|---|---|
| 绝杀模式下剩 1 秒出价是否有效？ | 状态机里隐式处理了 | → 提炼为显式规则：`倒计时归零前的出价均有效` |
| 延时模式连续出价会无限延长吗？ | 代码里有限制但没说明 | → 加上限：`最长延时不超原始时长 2 倍` |
| 封顶价到达后多余出价怎么处理？ | 直接返回成交拒绝 | → 规则：`封顶价到达后立即截标，后续出价返回已达封顶` |
| 出价金额精度（分 vs 元） | 项目用 `cents` 但容易混淆 | → 规则：`所有金额单位统一为分(int64)，前端展示时换算` |

**蒸馏流程**（每次 AI 犯错时自动触发）：

```
AI 生成了错误的出价校验逻辑
    ↓
人指出问题
    ↓
AI 反问："我是基于什么规则错的？正确的规则应该是什么？"
    ↓
人回答后 → 更新到 `docs/rules/auction-rules.yaml`
    ↓
后续产线配置引用该规则文件
```

**初始规则库**（`docs/rules/auction-rules.yaml`）:

```yaml
rules:
  - id: bid-amount-valid
    scenario: "用户出价"
    condition: "amountCents > currentPriceCents AND (amountCents - currentPriceCents) % bidIncrementCents == 0"
    action: "接受出价"
    exception: "若 amountCents >= capPriceCents，直接成交，不再校验加价步长"

  - id: auction-status-transition
    scenario: "状态流转"
    condition: "当前状态 + 操作必须在状态机允许的合法对中"
    action: "拒绝非法流转"
    exception: "取消操作（cancel）在 running 和 scheduled 状态下均可执行"

  - id: bid-idempotency
    scenario: "重复出价"
    condition: "idempotencyKey 在本次竞拍中已被使用"
    action: "返回已有出价结果，不重复执行"
    exception: "仅当 idempotencyKey 对应的出价是本次竞拍的最新出价时复用"

  - id: extension-timeout
    scenario: "延时模式"
    condition: "距 endAt 不足 extendThresholdSec 且未达最大延长时间"
    action: "endAt 向后延长 extendDurationSec"
    exception: "已超过最大延长时间（原始时长 * 2）或已达封顶价，不再延长"
```

### 硬骨头 2：偏差仲裁能力建设

**当前项目阶段不需要专职人员**，但你作为项目负责人需要储备以下两种能力：

**① 产线调优师（你兼任）**

你已有的能力：
- 懂 Go + React + TypeScript → 能判断 AI 输出质量
- 懂竞拍业务逻辑 → 能写校验规则
- 会用 Codex → 已经在用 AI 开发

你需要刻意练的：
- 把"感觉代码不对"翻译成"可自动验证的规则"
- 把"这个 bug 改完就完了"变成"这个失败案例→新规则→喂给 AI"

**实战练习**（本周做）：
1. 让 AI 生成一个"错误的出价检验"（故意漏掉封顶判断）
2. 你发现后，更新 `auction-rules.yaml` 的规则 1
3. 让 AI 重跑产线，确认不再犯同类错误
4. 把这次案例记录到 `docs/rules/deviations-log.md`

**② 认知边界守卫（未来的你或资深同事）**

只在以下场景介入：

| 介入场景 | 示例 |
|---|---|
| AI 自修复 3 次仍失败 | Lua 脚本死锁问题，AI 反复改但没抓到根因 |
| 涉及真实资金的风险判断 | 流拍后是自动补偿买家券还是人工介入？ |
| 模糊的业务需求定义 | "优化竞拍体验" → 你需要拆成可验证的指标 |
| 两个产线之间的冲突 | 出价产线说已成交，结算产线说还在 running |

每次介入后，必须产出**一条可验证的新规则**加入规则库，确保下次遇到同类问题 AI 自己能处理。

## 六、验证方式

框架落地的效果不靠"文档写了"来判断，靠以下可测量指标：

| 指标 | 当前基线 | 第 1 周目标 |
|---|---|---|
| AI 生成的代码通过一次测试的比例 | 约 70%（估算） | > 85% |
| 人 Review 代码的占比 | 100%（靠人看） | < 20%（靠测试看） |
| 规则库规则数 | 0 | > 15 条 |
| 产线配置数 | 0 | 3 条 |
| 新功能从需求到全量测试通过的周期 | 半天（手动） | 1 小时（AI 产线） |

---

> **核心思想**：你不是在"让 AI 帮你写代码"——你是在设计一条**认知流水线**：你的业务知识变成规则文件，AI 变成执行者，测试变成质量门。你从写代码的人，变成了设计产线的人。
