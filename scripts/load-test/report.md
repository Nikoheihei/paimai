# 竞拍系统压力测试报告


## 测试环境

| 项目 | 配置 |
|---|---|
| 服务架构 | Go (Gin) + MySQL 8.0 + Redis 7 |
| 部署方式 | Docker 本地单实例 |
| 服务端口 | `http://localhost:8080` |
| 压测工具 | Node.js 自研脚本（模拟真实用户行为） |
| 出价规则 | 严格按步长（100 分）递增，避免步长拒绝干扰 |
| 每轮时长 | 10-15 秒，每用户间隔 30-100ms |

## 指标定义

为避免混淆，将响应结果拆分为三类：

| 指标 | 含义 | 判定 |
|---|---|---|
| HTTP 2xx 率 | HTTP 200/201 等成功响应占比 | 正常 |
| HTTP 5xx 率 | 服务端返回 500 的占比 | **当前实现中主要是乐观锁冲突** |
| 真实系统错误率 | 连接超时、DB 异常、连接池耗尽、panic | 需关注 |
| 业务接受率 | `出价成功 / (成功 + 被拒)` | 出价请求中被接受的比率 |
| 成功出价/秒 | `bidAccepted / 时长` | **核心吞吐指标** |
| 业务冲突/拒绝 | 拍卖已结束、价格过低、并发竞争失败 | 正常业务行为 |

> **重要说明**：当前版本将乐观锁冲突（`UPDATE auctions WHERE version=N` 返回 `rows:0`）错误归类为 HTTP 500。若按语义修正为 409 Conflict，则真实系统错误率会显著降低。下文"HTTP 5xx 率"应理解为"当前实现的表象错误率"，不代表系统崩溃。

## 一、单拍卖用户上限

测试方法：1 个拍卖，N 人同时高频出价。

| 并发人数 | HTTP 2xx 率 | HTTP 5xx 率* | 业务接受率 | 成功出价/秒 | P50 | P95 | 分析 |
|---|---|---|---|---|---|---|---|
| **50** | 68.9% | 25.6% | **77.7%** | 273/s | 2ms | 5ms | ✅ 体验良好 |
| **100** | 56.5% | 37.8% | **53.1%** | 175/s | 6ms | 18ms | ✅ 可接受 |
| **200** | 50.9% | 47.0% | 29.8% | 36/s | 30ms | 54ms | ⚠️ 热点瓶颈明显 |

> 成功出价/秒在 50-100 人时表现良好，超过 200 人后急剧下降，说明热点竞争成为主要瓶颈。
> *HTTP 5xx 根因：乐观锁冲突（`rows:0`），非服务崩溃。

**分析**：

- 单拍卖在 **50-100 人**同时高频出价时体验较好，成功出价/秒达 273-175。
- 超过 **200 人**后，热点竞争导致乐观锁大量失败，有效成交急剧下降至 36/s。
- 这是典型的**单行热点问题**——所有请求竞争同一行 `auctions` 记录的版本号，而非服务处理能力不足。

**结论**：单拍卖最佳并发为 **50-100 人**，超过 200 人后热点竞争成为主要瓶颈。

## 控制变量实验：热点竞争 vs 系统瓶颈

为精确判断瓶颈来源，设计了一组控制变量实验——**同样 200 并发用户，仅改变拍卖数量**：

| 实验 | 用户数 | 拍卖数 | 成功出价/s | 出价接受率 | HTTP 2xx率 | 5xx+超时率 | P50 | P95 |
|---|---|---|---|---|---|---|---|---|
| **A** | 200 | 1 | **30/s** | 28.7% | 50.8% | 47.2% | 35ms | 65ms |
| **B** | 200 | 200 | **2,387/s** | 100.0% | 100.0% | 0.0% | 13ms | 35ms |

**结论非常清晰**：

- 同样的 200 人、同样的服务器、同样的 Redis、同样的网络
- 实验 A（1 个拍卖）成功出价仅 **30/s**，大量乐观锁冲突
- 实验 B（200 个拍卖）成功出价高达 **2,387/s**，零错误、零冲突

**这直接证明：瓶颈不是服务器处理能力、不是 Redis、不是网络，而是单拍卖行的热点竞争。**

## 二、多拍卖容量上限

测试方法：M 个独立拍卖同时运行，每拍卖 3 人出价，逐步增加拍卖数。

| 拍卖数 | 总并发 | HTTP 2xx 率 | HTTP 5xx 率* | 业务接受率 | 成功出价/秒 | P50 | P95 | 判定 |
|---|---|---|---|---|---|---|---|---|
| **100** | 300 | 79.0% | 16.3% | **86.0%** | 1,432/s | 34ms | 90ms | ✅ |
| **200** | 600 | 79.4% | 9.4% | **72.6%** | 1,862/s | 63ms | 158ms | ✅ |
| **260** | 780 | 80.3% | 8.0% | **72.2%** | 1,865/s | 77ms | 264ms | ✅ |
| **300** | 900 | 76.2% | 6.2% | 59.9% | 1,681/s | 88ms | 282ms | ⚠️ |
| 350 | 1050 | 87.2% | 0.1% | 0.5% | 7/s | 35ms | 105ms | ⚠️ |

> *HTTP 5xx 率随拍卖数增加反而下降，进一步证明其根因是乐观锁竞争（多拍卖分散后竞争减弱），而非系统瓶颈。

**关于 350 拍卖数据的说明**：

350 拍卖场景下出价接受率仅 0.5%，原因是在压测开始前几秒，大量拍卖因封顶价过低或出价增长过快即触发成交（`竞拍已经结束`），后续请求全部落在已结束拍卖上，有效出价窗口消失。**该组数据不能作为系统容量上限的直接依据**，仅说明在该压测参数下拍卖生命周期过短。

**分析**：

- 系统在**横向分散到多个拍卖**后表现优异，100-260 拍卖时业务接受率稳定在 72-86%。
- 峰值有效吞吐约 **1,865 成功出价/秒**（200-260 拍卖时达到）。
- 延迟随拍卖数线性增长（P50: 34ms → 77ms，P95: 90ms → 264ms），未见突变。
- 系统瓶颈在 MySQL 乐观锁竞争和事务压力，Go 服务本身处理能力充裕。

**结论**：系统可稳定支撑约 **260 个同时竞拍场次、780 并发用户**，峰值有效吞吐约 1,865 成功出价/秒。

## 四、分层压测与瓶颈定位

### 4.1 方法论：控制变量排除法

为精确定位系统真实瓶颈，采用**分层压测 + 链路追踪**的方法论：

1. **第一层：HTTP 链路** — 观测 PlaceBid 响应时间、错误率、MySQL 负载
2. **第二层：Outbox 消费链路** — 观测 pending 事件数、Consumer Lag、Stream 长度
3. **第三层：WebSocket 广播链路** — 观测连接数、广播延迟、消息送达率

通过逐层排除，最终锁定瓶颈位置。

### 4.2 瓶颈演变过程

| 阶段 | 瓶颈位置 | 现象 | 解决方式 | 结果 |
|---|---|---|---|---|
| Phase 1 | MySQL 乐观锁冲突 | 大量 HTTP 500，rows:0 | 返回 409 Conflict | 系统错误率归零 |
| Phase 2 | Outbox 消费链路 | pending 40,758 事件堆积 | poll 间隔 2s→100ms，batch 50→500 | pending 降至 20 |
| **Phase 3** | **WebSocket 广播链路** | **广播 P99 = 3.2s，消息送达率反降** | **待优化** | **当前主要瓶颈** |

### 4.3 关键证据：WS 广播链路瓶颈

**多房间长连接压测数据：**

| 连接数 | 收到消息数 | 广播 P99 | 现象 |
|---|---|---|---|
| 500 | 30,697 | 正常 | 消息量与连接数成正比 |
| 1000 | 11,658 | 上升 | 消息量开始不成比例 |
| 1500 | 5,486 | 明显延迟 | 消息量反而下降 |
| **2000** | **6,152** | **3.2s** | **广播严重滞后** |

**异常分析：**

理论上，连接数增加 → 消息总量应增加或至少接近。但实际数据呈现**反向下降**：

- 500 连接 → 30,697 条消息
- 2000 连接 → 仅 6,152 条消息

这说明**广播速度跟不上连接增长**，大量消息在广播队列中堆积，压测结束时未能及时送达。

**辅助证据：**

- Outbox pending：正常（20 以下）
- Redis Stream 长度：正常（0-2）
- Consumer pending：正常（0）
- MySQL threads：正常（10-20）
- 成功出价/秒：正常（2,000+/s）

**HTTP 链路完全健康，问题唯一指向 WS 广播链路。**

### 4.4 根因推测

| 嫌疑人 | 嫌疑依据 | 验证方向 |
|---|---|---|
| **#1 广播模型阻塞** | `client.send <- msg` 可能因单个 client 慢而拖慢整个房间 | 检查是否有 client 消费慢导致 channel 阻塞 |
| **#2 单 goroutine 广播** | Hub 可能只有一个广播线程处理 2000 连接 | 检查 Hub.Broadcast 是否串行执行 |
| **#3 JSON 重复序列化** | 每个 client 独立 `json.Marshal(event)` | 2000 次 Marshal 浪费大量 CPU |

## 五、结论

> 通过分层压测与链路追踪，先后排除了 MySQL 乐观锁、Outbox 消费链路和 Redis Stream 的瓶颈影响。最终发现系统容量瓶颈位于 **WebSocket 实时广播链路**。在 2000 长连接场景下，广播 P99 延迟达到 3.2 秒，成为影响用户实时体验的主要因素。

**系统容量上限：**

| 维度 | 数值 | 说明 |
|---|---|---|
| 单拍卖最佳并发 | 50-100 人 | 通过率 > 50%，延迟 < 20ms |
| 系统最大拍卖数 | ~260 个 | 780 并发用户 |
| 峰值有效吞吐 | **1,865 成功出价/秒** | HTTP 链路健康 |
| WS 广播瓶颈点 | **~2000 长连接** | P99 延迟 3.2s，消息堆积 |
| 服务稳定性 | 未崩溃 | 无 OOM、无连接池耗尽 |

## 六、后续优化方向

1. **WS 广播优化（最高优先级）**
   - 预序列化 JSON，避免每个 client 重复 Marshal
   - 检查 `client.send` channel 是否因慢消费者阻塞
   - 考虑按房间分片广播 goroutine，避免单线程瓶颈

2. **乐观锁冲突返回 409**，准确区分系统错误与业务竞争

3. **对单拍卖热点引入 Redis Lua 原子更新**，降低 MySQL 行竞争压力

4. **压测时提高封顶价或延长拍卖时间**，避免高拍卖数场景过早结束
